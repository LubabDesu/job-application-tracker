     Plan: T12 вҖ” Service Worker + MCP Client + Popup Status

     Context

     T11 (Workday + Greenhouse content scripts) is complete and confirmed
     working. Content scripts detect job applications and broadcast JOB_DETECTED
      messages via chrome.runtime.sendMessage. T12 wires up the receiving end: a
      service worker that calls the MCP server, plus a popup that shows live
     status so the user can see what's happening.

     Architecture

     Content Script вҶ’ chrome.runtime.sendMessage({type:'JOB_DETECTED', job})
       вҶ’ Service Worker (onMessage handler)
         вҶ’ reads settings from chrome.storage.sync
         вҶ’ callLogApplication(mcpUrl, mcpSecret, job)  [mcp-client.ts]
           вҶ’ POST {mcpUrl}/log with Bearer auth + job JSON
             вҶ’ MCP server /log endpoint вҶ’ handleLogApplication вҶ’ Notion
         вҶ’ stores {company, role, jobId, status, loggedAt, error?} in
     chrome.storage.session
           вҶ’ Popup reads chrome.storage.session вҶ’ shows live status

     Why a /log REST endpoint (not raw MCP JSON-RPC)

     The MCP server uses StreamableHTTPServerTransport which requires a full
     JSON-RPC handshake (initialize вҶ’ tools/call). Implementing that in a
     service worker is complex and fragile. A POST /log REST shim on the MCP
     server gives the extension a clean, direct interface while keeping MCP
     tools intact for AI clients (Claude Desktop etc.).

     What Changes

     1. mcp-server/src/transports/http.ts вҖ” Add /log REST endpoint

     Add logApplication callback param and handle POST /log:

     export function startHttpTransport(
       createMcpServer: () => McpServer,
       logApplication: (args: LogApplicationArgs) => 
     Promise<LogApplicationResult>,
       port: number,
       secret: string
     ): void

     POST /log: read body вҶ’ call logApplication(args) вҶ’ respond with
     LogApplicationResult JSON.
     Import LogApplicationArgs and LogApplicationResult from
     ../tools/log-application.js.

     3. extension/src/shared/mcp-client.ts вҖ” Implement callLogApplication()

     Replace stub with fetch to {mcpUrl}/log:
     - Authorization: Bearer {mcpSecret}, Content-Type: application/json
     - Body: {company, role, url, jd_text: job.jdText, source_platform: 
     job.sourcePlatform, location?}
     - Returns {success: true, jobId} on 200, {success: false, error} otherwise
     - Catches network errors (fetch throws) and JSON parse failures

     4. extension/src/background/service-worker.ts вҖ” Full implementation

     chrome.runtime.onMessage.addListener((message, _sender, sendResponse) => {
       if (message.type !== 'JOB_DETECTED') return false
       handleJobDetected(message.job as DetectedJob)
         .then(() => sendResponse({ ok: true }))
         .catch((err: unknown) => sendResponse({ ok: false, error: String(err)
     }))
       return true  // keep channel open for async sendResponse
     })

     handleJobDetected(job: DetectedJob):
     1. Read ExtensionSettings from chrome.storage.sync (fallback:
     DEFAULT_SETTINGS)
     2. If settings.mcpSecret is empty вҶ’ store {status: 'error', error: 'MCP 
     secret not configured', company, role, loggedAt} in session, return
     3. Store {status: 'pending', company, role, loggedAt} in session (popup
     shows "Logging...")
     4. Call callLogApplication(settings.mcpUrl, settings.mcpSecret, job)
     5. On success: update session to {status: 'logged', company, role, jobId, 
     loggedAt}
     6. On error: update session to {status: 'error', company, role, error, 
     loggedAt}

       - Pending: spinner + "Logging {role} at {company}..."

  What just happened вҖ” explained simply:

  The extension has three separate JavaScript environments that can't directly
  call each other:

  Content script (workday.ts) вҖ” runs inside the Workday webpage. It watches the
  DOM for the confirmation screen, then calls chrome.runtime.sendMessage({ type:
   'JOB_DETECTED', job: {...} }) вҖ” this is like shouting a message into the
  extension's internal radio channel.

  Service worker (service-worker.ts) вҖ” a background process Chrome runs for the
  extension. It listens on that same radio channel with
  chrome.runtime.onMessage.addListener(...). When it hears JOB_DETECTED, it
  fetches chrome.storage.local to get your MCP server URL and secret, then POSTs
   to http://localhost:3000/log to log the job to Notion.

  chrome.storage is just a key-value store built into Chrome for extensions.
  Three flavors:
  - .local вҖ” stored on your machine only (we use this for settings like
  mcpSecret)
  - .sync вҖ” synced to your Google account across devices (we avoided this for
  secrets)
  - .session вҖ” wiped when Chrome closes (we use this for the "last logged" popup
   state)

  Why the smoke test worked: You sent the JOB_DETECTED message from the popup
  console (an extension context) вҶ’ service worker received it вҶ’ read mcpSecret
  from .local вҶ’ POSTed to your running MCP server вҶ’ Notion row created. The
  popup picked up the result because the service worker wrote to .session and
  the popup's onChanged listener fired.

  The gray area: The content script fires JOB_DETECTED only when it detects
  confirmation page text like "Thank you for applying" вҖ” that detection needs to
   be verified on a real Workday application flow.

  The Three Worlds

  When you load a webpage, Chrome creates a renderer process for it вҖ” an

  ---
  World 1: Content Script

  Injected by Chrome into a matching webpage. It runs alongside the page's JS
  but in a separate JavaScript scope вҖ” they don't share variables, functions, or
   window objects. The content script can read and modify the DOM because it
  shares the same HTML document, but it can't call Workday's JavaScript
  functions and Workday can't call yours.

  Page JS:        window.myVar = 'hello'
  Content Script: console.log(window.myVar)  // undefined вҖ” different window
  object

  This isolation is intentional вҖ” it prevents malicious pages from hijacking
  your extension.

  What it CAN do:
  - Read/manipulate the DOM (document.querySelector(...))
  - Listen for DOM mutations (MutationObserver)
  - Call a limited set of chrome.* APIs вҖ” mainly chrome.runtime.sendMessage
  - Make fetch requests (subject to CORS)

  What it CANNOT do:
  - Access other tabs
  - Persist data long-term (no file system, localStorage is the page's,
  chrome.storage is blocked on strict CSP pages)
  - Keep running after the tab closes

  In this project: workday.ts runs here. It watches for the confirmation screen
  and calls chrome.runtime.sendMessage({ type: 'JOB_DETECTED', job }).

  ---
  World 2: Service Worker

  This is the extension's brain. It's a background script that Chrome runs
  independently of any tab.

  Key difference from old MV2 "background pages": it's not always running.
  Chrome kills it after ~30 seconds of inactivity to save memory, then wakes it
  up when an event fires (a message arrives, an alarm triggers, etc.). This
  means:

  - Any in-memory variables you set will be gone when it wakes up next time
  - You must persist anything important to chrome.storage before the worker
  sleeps
  - It has no DOM вҖ” can't touch document at all
  - It CAN use fetch, chrome.tabs, chrome.storage, everything except DOM APIs

  In this project: service-worker.ts runs here. It wakes up when JOB_DETECTED
  arrives, reads settings from chrome.storage.local, calls the MCP server,
  writes the result to chrome.storage.session.

  ---
  World 3: Popup

  A mini browser window that opens when you click the extension icon. It's just
  a normal webpage (index.html + React) that happens to have access to all
  Chrome APIs.

  It only exists while open. If the user closes it, the entire React component
  tree is destroyed вҖ” state is gone. That's why we store the last-logged job in
  chrome.storage.session rather than React state: the popup reads it fresh every
   time it opens.

  In this project: App.tsx runs here. On mount it reads chrome.storage.session
  to show the last result, then subscribes to chrome.storage.onChanged so it
  updates live if the service worker writes a new result while the popup is
  open.
